# Review Docs Command

Review and update Foam documentation, separately for user-facing docs and developer docs, to ensure accuracy, completeness, and consistency.

## Usage

```
/review-docs [target]
```

## Parameters

- `target` (optional): Which docs to review — `user`, `dev`, or omit for both

## Description

Documentation rots over time: files reference deleted targets, stubs linger with no content, proposals describe things already built, and features go undocumented. This command audits and improves the docs systematically.

Run with a plan-first approach:
1. Survey the doc tree
2. Identify problems
3. Present findings and proposed actions to the user
4. Get confirmation before making changes
5. Execute changes
6. Verify no dangling references remain

## User Docs (`docs/user/`)

Audience: non-technical users new to Foam. Docs must be concise, task-oriented, and example-driven.

### What to check

1. **Stubs** — files whose entire content is a placeholder ("This item needs more specification work", "TODO", etc.). Either delete them or fill them in.
2. **Stale content** — references to removed features, outdated tool names (e.g. Gatsby templates), or instructions that no longer match the UI.
3. **Missing coverage** — features that exist in the codebase but have no user doc. Check `src/features/` against `docs/user/features/`.
4. **Orphaned proposals that graduated** — if a proposal describes a feature that is now implemented, merge the useful parts into the user doc and delete the proposal.
5. **Dead link refs** — autogenerated `[//begin]...[//end]` blocks pointing to deleted files. Remove the ref and the inline usage.
6. **Navigation gaps** — new or updated docs not linked from `docs/user/index.md`, `recipes.md`, or relevant getting-started pages.

### Writing guidelines

- Lead with what the user can do, not how it works internally
- Keep it short: one concept per section, no padding
- Use examples over abstract descriptions
- No Roam/Obsidian comparisons unless directly useful to the reader
- No `[[todo]]` markers pointing to deleted files

## Dev Docs (`docs/dev/`)

Audience: contributors and maintainers. Docs should capture decisions, constraints, and architecture — not meeting notes or timelines.

### What to check

1. **Obsolete proposals** — proposals describing things that are now built (e.g. `foam-core.md` after `foam-core` was extracted). Delete them.
2. **Stubs with no content** — delete or merge into a more appropriate file.
3. **Implemented designs** — if a proposal has been fully implemented, the useful parts (design rationale, constraints, architecture) should move to a stable dev doc; the proposal itself can be deleted.
4. **Naming** — the `proposals/` folder implies tentative status. Use `design/` for active architecture docs.
5. **Roadmap hygiene** — remove entries that point to deleted files; remove `[[todo]]` links pointing to deleted `todo.md`.

### What to keep

- Active architecture and design docs (e.g. publishing pipeline plans)
- Contribution guide, code of conduct, release process
- Anything a new contributor needs to understand the codebase

## Handling Proposals

When reviewing `docs/dev/design/` (formerly `proposals/`):

- **If fully implemented**: extract useful rationale into a stable dev doc or user doc, then delete the proposal
- **If partially implemented**: note which parts are done, update the doc to reflect current state, keep only what's still forward-looking
- **If never started and no longer relevant**: delete
- **If actively planned**: keep as-is in `design/`

## Checking for Dangling References

After any deletion or rename, always run a search for references to removed files across the entire `docs/` tree:

```bash
grep -r "deleted-filename" docs/ --include="*.md"
```

Fix every hit: remove the inline usage and its link ref from the `[//begin]...[//end]` block.

## Output

- A list of findings grouped by severity: **delete**, **update**, **merge**, **missing**
- Proposed actions for each finding, presented to the user before execution
- A final confirmation that no dangling references remain
